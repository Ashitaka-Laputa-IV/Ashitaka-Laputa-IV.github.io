上下文折叠

这是我最近发现自己在做的一个有趣动作：

  1. 遇到复杂的编程挑战

  2. 获取三篇与我问题相关的论文

  3. 打开三个终端窗口，让不同的 Claude 阅读每篇论文

  4. 对每篇论文问一些问题

  5. 将每次对话的摘要保存到 .md 文件。

  6. /clear

  7. 让 Claude 阅读所有三个摘要，生成一个带有比较表的综合分析。

  8. 保存到 synthesis.md。

  9. /clear

  10. 包含 @synthesis.md 并与 Claude 头脑风暴

  11. 一个解决方案出现

这对学习速跑也非常有效。

  1. 去 plato.stanford.edu 并获取三个不同哲学家的页面（柏拉图、康德、尼采，等等）

  2. 打开三个终端窗口，让不同的 Claude 阅读每页

  3. 对每位哲学家问一些问题

  4. 将每次对话的摘要保存到 .md 文件。

  5. /clear

  6. 让 Claude 阅读所有三个摘要，综合、比较和对比。

  7. 保存到 synthesis.md。

  8. /clear

  9. 包含 @synthesis.md 并进行对话

（Claude Code 是一个优秀的思维工具，顺便说一下。）

我认为这个动作是上下文折叠，因为它感觉像烘焙时折叠面团。当我折叠上下文时，我从 AI 获得的结果要好得多，好得多。

为什么折叠上下文如此有效？嗯，这几乎就像我们在手动进行深度研究。每个 Claude 实例都作为研究代理发挥作用。关键是，这些是我们手动指导的研究代理。在每个线程中，我们都在提出问题，提取出对我们更大问题最有意义的上下文维度。这是一种智能压缩，而压缩 = 智能力，或者足够接近。

上下文折叠也创造了有用的副产品。这些小 markdown 文件是有价值的压缩洞察，你可以在未来的对话中包含它们，用深层上下文背景来注入它们。我们的团队已经开始将它们检入代码库下的 docs 目录中。

上下文折叠是从 LLM 引发有效行为的一般模式的一个例子。当我们折叠上下文时，我们发散，在不同的上下文窗口中探索不同的路径。在探索解决方案空间后，我们收敛，比较我们学到的东西，并将所有内容综合在一起。

Claude Code 的计划模式是这个模式的另一个例子。首先 Claude 发散，用子代理探索代码库，然后 Claude 收敛，编写一个综合发现结果。

[

](https://substackcdn.com/image/fetch/$s_!Ppiq!,f_auto,q_auto:good,fl_progressive:steep/https%3A%2F%2Fsubstack-post-media.s3.amazonaws.com%2Fpublic%2Fimages%2F7e2da523-34a8-4caa-9dc7-fa70fda082ac_700x500.jpeg)

设计师称这个过程为双钻模型。当使用双钻模型时，我们经常做两次，一次用于研究，一次用于开发。今天大多数 AI 工具只做一次（计划模式）。做多次是否有收益？

控制论学者 Horst Rittel 将双钻模型置于更大的理论框架中。他说设计是"多样性的生成和多样性的减少。"

这里的多样性意味着控制论多样性，即系统可以生成的所有可能行为的集合。

Ashby 的必要多样性定律： 调节器的多样性必须至少与它调节的系统的多样性一样大。

Ashby 定律告诉我们，解决方案必须至少与问题一样复杂。如果不是，问题会溢出，绕过解决方案，成为外部性。这是有用的信息。它告诉我们，要解决问题，LLM 必须至少与问题一样大。

而 LLM 确实有很多控制论多样性。一个互联网的价值。这种极端多样性在你运行原始基础模型时显现。你得到的是非人类的、某人通道超维度宇宙恐怖的咆哮。基础模型的多样性远远超出了正常人类行为的范围。

[

](https://substackcdn.com/image/fetch/$s_!ct-P!,f_auto,q_auto:good,fl_progressive:steep/https%3A%2F%2Fsubstack-post-media.s3.amazonaws.com%2Fpublic%2Fimages%2F526fe074-8061-4a33-8e00-551e53cace06_1007x790.png)

所以，LLM 有很多多样性。太多的多样性无法适应人类含义的微小参考框架。解决方案？在顶部贴一个笑脸。RLHF 将 LLM 的多样性减少到有意义的范围。

但 RLHF 几乎太多地减少了多样性。模型带有一个强烈的倾向去寻求燕麦片。它已经失去了必要的多样性来惊喜。

但处于控制中意味着定义将被考虑的范围，即可能的范围。实际上，当我处于控制中时，我将世界限制在我可以想象或允许的范围内……
(Ranulph Glanville，2019，"设计控制论"）

我们可以通过上下文窗口带回一些这种多样性。LLM 通过上下文窗口高度可引导，通过提示，你可以从模型中排序或挑选潜在维度，提取出不同的行为。

这就是为什么将 LLM 置于与工具的循环中让它变得如此聪明。工具向上下文窗口添加上下文多样性，这从模型中引出潜在多样性。模型和外部上下文之间的对话产生了有意义的行为，具有多样性来惊喜。

OpenClaw 是这里的一个极端例子。当你给爪子完全访问你的计算机时，它将高级图灵机的多样性导入代理循环。那是很多多样性，潜在无界。其中一些多样性是深刻惊喜的，比如当 OpenClaw 的创建者发现它可以自己弄清楚如何做语音转录时。其中一些多样性是负面惊喜的，比如当你要求你的爪子整理你的收件箱，它删除了你所有的电子邮件。无论哪种方式，具有完整 bash 访问的爪子都有足够多的多样性来惊喜。

没有足够多样性来控制系统的巨大好处是，如果我放弃试图控制，我可以发现如果我坚持处于控制中会排除的许多可能性。这些可能性是意外的，在我的参考框架之外，用一个词来说，新颖的。
(Ranulph Glanville，2019，"设计控制论"）

从必要多样性的角度来看，折叠上下文很有道理。我们让我们的代理的理解发散、专业化，并向意外方向旋转。他们理解中的差异、矛盾，是阿尔法，有价值的多样性。
