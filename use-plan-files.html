使用 .plan 文件

Ashitaka

周五 27 二月 2026

.plan 文件

为什么

首先，这很简单：一个纯文本文件，很少的语法规则，公开可用，就是这样。

其次，保持我的 .plan 文件更新帮助我保持有序。你是否遇到了想要修复但现在没有时间做的事情？为此添加一个条目。你是否发现了一篇有趣的文章想要……稍后阅读？把它添加到你的 .plan 文件中。你是否花了大量时间调试一个问题？把它写下来，以便以后如果发生类似的事情可以回来查看。这就像你的数字个人日记。

最后，跟上你的 .plan 文件将有助于提高你的技术写作技能，特别是如果你和我一样在同一个行业工作，你知道这有多重要（例如，记录事情，提交清晰和描述性的错误报告，或描述技术挑战的解决方案）。

格式

我的 .plan 文件的格式非常基础：主要基于 John Carmack 早期的 .plan 文件，只是加入了一点 Markdown 风味。

通常，我所有 .plan 文件的前几行会告诉你这个文件是关于什么的，并给出文件本身结构如何的一点概述。例如：

这是我的 .plan 文件 ...

当我完成某事时，我在那一天写一行。每当在一天中提到一个错误/缺失功能/想法而我没有修复它时，我会记下来并用 ? 标记它。有些东西在修复之前会被记录很多次。偶尔我会回顾旧笔记并用 + 标记我后来修复的东西，用 ~ 标记我后来失去兴趣的东西。

有时我会抱怨坏掉的东西（主要是与计算机相关的），以及我如何（尝试）修复它们。

--- Matteo Landi

细心的读者可能会注意到，这个欢迎消息与 John Carmack 在 90 年代的 .plan 文件中的内容非常相似（90% 相似）；只是做了一些调整，使文件更加 Markdown 友好（用 ~ 代替 - 来表示我不再感兴趣的东西，或者用 ? 前缀我的待办事项）。我没有明确说明每日条目是如何实现的，但至少读者知道会有一些（一旦他们浏览文件，理解结构应该非常简单）。

说到每日条目，在我的 .plan 中，这些是作为 Markdown 的章节标题实现的（其中包含特定日期），那天发生的任何事情都会放在它下面：

# 2020-05-12
做了一些调试，弄清楚为什么**所有**我的 Common Lisp Windows 可执行文件都没有按预期在 stdout 上打印东西，这似乎与重定向有关...
...

# 2020-05-16
好吧，我想我弄清楚了为了让我的 CL 程序在 Windows 上正确地将东西输出到 stdout 我需要做什么。原来你必须明确告诉 deploy[1]，我用来部署独立 CL 应用程序的系统，你希望你的 Windows 应用程序是一个 _console_ 应用程序，否则 deploy 会选择 GUI 应用程序（而 GUI 应用程序通常没有标准输出可写）。
...

如果对于给定的每日条目，我最终写的是完全不相关的话题，我会简单地在中间插入一个 --- 分隔符，并创建两个或更多子条目：

# 2020-05-16
好吧，我想我弄清楚了为了让我的 CL 程序在 Windows 上正确地将东西输出到 stdout 我需要做什么。原来你必须明确告诉 deploy[1]，我用来部署独立 CL 应用程序的系统，你希望你的 Windows 应用程序是一个 _console_ 应用程序，否则 deploy 会选择 GUI 应用程序（而 GUI 应用程序通常没有标准输出可写）。

[1] https://github.com/Shinmera/deploy

    /home/travis/.travis/functions: eval: line 109: unexpected EOF while looking for matching `'
    /home/travis/.travis/functions: eval: line 110: syntax error: unexpected end of file
...

这就是你处理我的 .plan 文件需要知道的所有内容；了解一点 Markdown 不会有什么坏处（例如，强调、加粗、代码、链接），但 Markdown 足够简单，你可以在阅读时自己弄清楚。

个人设置

我不只有一个 .plan 文件，而是有多个。例如，我的个人文件（也可以在这里公开获得），是我写关于我必须完成（或已经完成）的事情、我开始工作的项目、想法，或我正在处理的与计算机相关的问题（以及一旦我弄清楚它们的解决方案）的地方。然后我有另一个我在工作中使用的 .plan 文件：与我的个人文件非常相似，但都是关于工作相关的事情。在工作中，我还为每个在我团队工作并向我汇报的人保留不同的 .plan 文件，主要用于 1-1 会议，或这些人在多年间完成的重要成就/项目。

我所有的 .plan 文件都存储在 Dropbox 中，由于我在我关心的所有 PC 上都安装了 Dropbox（甚至在我的手机上），我可以从任何地方编辑我所有的 .plan 文件。当然，你不一定非要使用 Dropbox，但我发现它太方便了，以至于不使用它：在移动中编辑你的 .plan 文件，点击保存，当你回到 PC 时，添加的内容会在那里；或者反过来，你关闭 PC，上火车，你在手机上打开 Dropbox，几秒钟后你就可以从之前离开的地方继续工作。当然，我没有每次在移动中时都编辑我的 .plan 文件的冲动，但如果我必须，我可以这样做。

最后，我在我的服务器上有一个 cron(8) 任务，每晚下载我的 .plan 并将其放入特定文件夹，以便我的 Web 服务器可以提供它。我还写了一个小程序，plan-rss，我的服务器自动运行它来创建我的 .plan 文件所有最新每日条目的 RSS 源（它并不完美，但如果你正在寻找一种跟上我的 .plan 文件的方法，那么订阅我的 plan.xml 可能是你最好的选择）。

结束思考

这就是我关于 .plan 文件要说的所有内容，如果有的话，我希望我至少让你对它们背后的想法感到一点好奇。

记住：你最终使用哪种语法规则、你将在哪个编辑器中输入这些笔记，或者你最终将支持哪种标记语言规则并不重要。真正重要的是你开始写作，开始记笔记……更重要的是，你一致地这样做：你做得越多，它就会感觉越自然，而在花了这么多时间在电脑前后，一天结束时没有在 .plan 文件中创建条目，这会感觉越奇怪。

愉快地使用 .plan 文件吧
