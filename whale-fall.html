鲸落

Ashitaka

周五 27 二月 2026

当鲸鱼在开放海域死亡时，它的尸体沉入深渊海底并成为一个生态系统。海洋生物学家称之为鲸落，尸体在三个重叠阶段维持生命：移动清道夫在几个月内剥离软组织，富集机会主义者在数年内定殖骨骼和周围沉积物，化学合成细菌在数十年内以骨骼本身为食，将骨骼中储存的脂质转化为支持整个专门生物群落的能量。一个单一的鲸落可以在原本贫瘠的海底维持生命五十年。

Michael Winser 在我们谈论废弃项目的依赖图会发生什么时，顺便提到了鲸落，这让我无法忘怀。

一个大型开源项目变得无人维护。也许维护者精疲力竭，也许背后的公司转向。项目停止获得更新但没有消失。它坐在 GitHub 上积累问题，它的最后一次提交进一步退入过去，有人分叉它开始合并最紧急的补丁。如果项目足够受欢迎，会出现多个分叉，为用户竞争，就像盲鳗为鲸脂竞争一样，尽管大多数很快死亡，一两个依靠有足够时间或机构支持来继续前进的人的力量生存下来。OpenOffice 以这种方式变成了 LibreOffice，MySQL 变成了 MariaDB，Hudson 变成了 Jenkins，每个都是通过熟悉的分叉公告、迁移指南和"为什么你应该切换"博客文章序列成长为规范替代品的清道夫分叉。

然后，结构骨架、协议和文件格式以及 API 契约，继续支持可能甚至不知道骨骼来自哪里的专门社区。OpenDocument Format 已经超过了创建它的 OpenOffice 社区，由跨数十个语言生态系统的文档格式库维持。Docker 在 2015 年将其容器运行时和镜像格式捐赠给开放容器倡议。OCI 规范现在定义了容器如何工作，无论运行时如何。Docker 自己的主导地位消退；规范没有。Tree-sitter 是为 Atom 构建的，在 GitHub 归档 Atom 后，它成为了 Zed、Neovim、Helix 和过去几年发布的大多数编辑器中的语法引擎。

演替

我不断注意到无人维护的库的模式是连续的重新定殖。一个项目变得安静，有人分叉它，其他项目开始依赖分叉，然后那个分叉维护者也精疲力竭，整个循环在较小规模上重复。每一代分叉通常比上一代更小，贡献者更少，用户基础更窄，直到最终想法本身迁移而不是代码。另一个语言生态系统中的某人看着积累的残骸，决定从头重写概念，向前携带设计但留下实现。

Sass 经历了这个。原始参考实现是一个 Ruby gem。当 Ruby 的性能成为瓶颈时，LibSass 用 C++ 重写了它，sassc gem 为 Ruby 用户包装了它。然后 LibSass 本身被弃用以支持 Dart Sass，后者现在是规范实现。概念在十年间从 Ruby 迁移到 C++ 再到 Dart，每次重写都受益于其前身的累积错误报告和设计论证，在每个阶段都有其他生态系统中的包装库以 Sass 语言规范的结构骨架为食。今天写 Sass 的大多数人不知道它开始时是一个 Ruby gem。

连续的重新定殖有一个糟糕的失败模式。Edera 发现了 Rust 的 tar-rs 库中的一个差异解析错误，影响了每个下游分叉：tar-rs 本身、async-tar、tokio-tar，以及由像 Astral 这样的公司维护的多个内部分叉（其分叉随 uv 包管理器一起发布）。协调披露意味着在分叉的分叉图中联系大约二十个实体，其中四个库版本中有三个无人维护，几个维护者无法联系。漏洞存在是因为代码通过连续分叉被向前复制，而没有人重新审计所有分叉都从原始版本继承的 PAX 头解析。错误一直坐在骨骼中整个时间，被每个分叉继承。发现一个包的哪些分叉受公告影响是我正在单独处理的问题，因为目前没有人有好的工具来处理它。

CentOS 在 Stream 转向之后是操作系统规模的相同模式：Rocky Linux 和 AlmaLinux 分叉了，更小的 RHEL 兼容重建在它们周围的富集层中出现，下面的结构骨架、RPM 打包、systemd 约定、文件系统层次，无论特定发行版在任何时刻是活着还是死的，都保持不变。

许可证变更

当一个项目从开源许可证切换到源可用许可证时，清道夫阶段几乎立即触发，通常在变更甚至生效之前。Redis 到 Valkey，Elasticsearch 到 OpenSearch，Terraform 到 OpenTofu，模式现在足够熟悉，社区已经把它变成了常规：匆忙分叉最后一次开源提交，短暂竞争，并围绕一两个幸存者整合。在这些情况下，生物体并没有完全死亡。Redis 产品仍然有收入和路线图。但从开源生态系统的角度来看，代码体已经停止接受外部贡献，分叉开始从原始版本漂移，就像 MariaDB 从 MySQL 漂移一样。

抽象层是持续存在的部分。每个与开源版本集成的项目都面临跟随专有版本或切换到分叉之间的选择，其中许多只是构建兼容性垫片。这些垫片倾向于超过创建它们的争议，在许可证辩论冷却多年后，安静地以原始 API 的骨架为食。

Sun Microsystems

Oracle 在 2010 年收购 Sun 不如单一鲸落，不如整个鲸群同时在海上死亡。Java、Solaris、ZFS、DTrace、VirtualBox、NetBeans、GlassFish、Hudson、MySQL，每个都沉入单独的海底并产生自己的演替。一些产生了单一的主导分叉（Hudson 到 Jenkins，ZFS 到 OpenZFS），其他分散到竞争血统（MySQL 单独喂养了 MariaDB、Percona，以及短暂的 Drizzle，后者在被放弃时本身变成了一个较小的鲸落），一些在定居之前在基金会之间弹跳（NetBeans 到 Apache，GlassFish 到 Payara 和更广泛的 Jakarta EE 生态系统）。所有这些下面的结构骨架、JVM 字节码格式、ZFS 磁盘格式、MySQL 线协议，在开发人员从未听说过 Sun 的项目中仍然是承重的。

浅水

一些项目在浅水中死亡，那里尸体被快速回收。收购就是这样工作的：公司被吸收，代码变成专有或被归档，知识随着人们分散而不是积累在其他人可以喂养的公共尸体中。企业整合有类似的效果，因为当一个大型独立项目被折叠到平台公司的专有服务中时，营养在水柱中回收而不是下沉得足够深以让演替发生。

我认为当前在云提供商下的整合趋势正在减少开源中的鲸落率，这对生态系统多样性有二阶效应，没有人追踪。你可以测量它：查看随时间推移的废弃项目的分叉和依赖图，计算有多少新项目引用一个死依赖，比较 npm 与 crates 与 rubygems 中鲸落的半衰期。一些生态系统有更深的海洋底、更慢的分解、更持久的结构影响吗？数据存在于包注册表和 forge API 中，但我没有看到任何人问这个问题。

一个开源生态系统，其中每个大型项目都由平台公司拥有，无限期维护或在死亡时被安静吸收，是那些富集和化学合成阶段很少有机会发展，而依赖鲸落为食的小型专门生物从来没有机会演化的生态系统。最健康的生态系统有稳定的鲸落供应，这是一个奇怪的事情来支持，因为它意味着希望大型项目的死亡，除了深海海底没有其他食物来源。
